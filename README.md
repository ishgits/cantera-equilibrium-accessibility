# Cantera Equilibrium Accessibility Workflow

A reusable [Cantera](https://cantera.org/) workflow for one question:

> Given a starting inventory and a list of candidate products, which products are
> **thermodynamically equilibrium-accessible** under each modeled condition?

You supply species thermodynamic data (extracted from CHNOSZ or seeded from a CSV)
and a set of scenarios. The workflow fits NASA9 polynomials, generates one
single-product Cantera model per (scenario × candidate product), runs equilibrium
at the temperatures you choose, and reports whether each product appears above a
reporting threshold — plus supporting diagnostics, tables, and figures.

This repository ships with **no proprietary species data**. It includes a small,
fully reproducible example built entirely from public CHNOSZ values so you can
verify your setup end to end.

## What "single-product mode" means (and its main caveat)

Cantera will form *any* species present in a model file, regardless of its initial
amount. To keep the question unambiguous, the workflow builds a **separate model
file per candidate product**: the scenario's starting inventory plus exactly one
candidate. Each run answers "is *this* product accessible from this inventory?"

Because candidates never share a model, **they do not compete with one another**.
For a single-reactant scenario this makes the result effectively all-or-nothing:
the reactant either converts to the candidate (if more stable) or it does not.
The reported mole fraction reflects stoichiometric dilution, **not** a ranking of
products against each other. This is a deliberate design choice — see
[`docs/USAGE.md`](docs/USAGE.md#interpretation) for the full interpretation guide.

This approach is known as weak coupling. A justification and further discussion of
the underlying science are given in Madan & Pearce (2025, 2026); see
[References](#references).

This workflow reports **thermodynamic accessibility only**. It does not predict
kinetic rates, reaction pathways, experimental detectability, or synthesis yields.

## Quickstart

```bash
# 1. Core Python dependencies
pip install -r requirements.txt

# 2. Cantera (required to run equilibrium). conda-forge is the reliable route:
conda install -c conda-forge cantera

# 3. Offline sanity check — no Cantera or pyCHNOSZ required
python scripts/validate_static_workflow.py

# 4. Run the example end to end
jupyter lab notebooks/01_full_cantera_equilibrium_workflow_v4.ipynb
#    then Run All. The shipped Cell 1 is configured for the validation example
#    and seeds Gibbs data from inputs/example_validation_gibbs.csv (no pyCHNOSZ).
```

The example should report **adenine** and **cytosine** as accessible
(`formation_call = significant`) for the validation scenario at 0 °C, with all
outputs written under `data/results/` and `figures/`.

## Running your own study

1. Build a species metadata CSV (copy `inputs/species_template.csv`).
2. Build a scenarios YAML (copy `inputs/scenarios_template.yaml`).
3. Provide Gibbs-energy data by **one of two paths**:
   - seed from a wide CSV (`SEED_CACHE_FROM_EXISTING_WIDE_CSV = True`), or
   - extract from CHNOSZ (`RUN_CHNOSZ_EXTRACTION = True`, requires pyCHNOSZ).
4. Point Cell 1 of the notebook at your files and Run All.

> **Important seeding caveat:** `pyCHNOSZ` can only extract species that exist in
> the CHNOSZ database. Estimated or custom species (e.g. QC-derived Gibbs curves)
> **must** be supplied via the seed-CSV path; CHNOSZ extraction will fail for them.

Full instructions — every CSV column, every scenario key, every output table, and
how to read the results — are in [`docs/USAGE.md`](docs/USAGE.md).

## Sensitivity landscapes (`studies/`)

The base notebook answers "is product X accessible?" for one scenario. The
**sensitivity layer** sweeps many cases — starting-inventory grids and exact
Gibbs-energy (ΔG) offsets — to map *where* a product becomes accessible and how
robust that is. Each study is self-contained under `studies/<study_id>/`, and the
**only file you edit is `studies/<study_id>/study_config.yaml`** (heavily commented).

- **New to the sensitivity layer? Start here (no Cantera needed):** open
  [`notebooks/00_start_here_sensitivity_landscapes.ipynb`](notebooks/00_start_here_sensitivity_landscapes.ipynb)
  and **Run All**. It reads the committed alanine MVP outputs and teaches how to
  interpret the landscapes without running any equilibrium calculations.
- **Notebook (recommended for running studies):** open
  [`notebooks/02_sensitivity_landscape_workflow.ipynb`](notebooks/02_sensitivity_landscape_workflow.ipynb),
  set `STUDY_CONFIG` in Cell 1, and **Run All**. It is a thin viewer over the same
  engine the CLI uses, so the two cannot diverge.
- **CLI (same engine, for batch/headless):**
  ```bash
  python scripts/new_study.py --id my_study            # scaffold a study folder
  python scripts/run_sensitivity_study.py --config studies/my_study/study_config.yaml --dry-run
  python scripts/run_sensitivity_study.py --config studies/my_study/study_config.yaml
  python scripts/summarize_sensitivity_study.py --config studies/my_study/study_config.yaml
  python scripts/plot_sensitivity_study.py --config studies/my_study/study_config.yaml
  ```

Runs are resume-safe (`--only-failed`, `--force`, `--rebuild-coefficients`), reuse one
Cantera model per unique species/thermo set, and write tidy, ML-ready tables plus a
column dictionary (`results/SCHEMA.md`). The bundled `studies/alanine_mvp/` is a
worked example (625-point inventory grid + ΔG sweeps) and **ships with its generated
outputs committed** (design matrix, manifests, results, figures, provenance) so you
can browse a complete study without running anything. The amino-acid breadth
campaigns below are likewise committed with their outputs; only transient scan
scaffolds (`studies/amino_acid_scan/`) stay gitignored.

**Breadth campaign (one study per amino acid).** `scripts/new_amino_acid_batch.py`
scaffolds a study for each of the 18 C/H/N/O amino acids over a shared
CHNOSZ-derived dataset; `scripts/run_amino_acid_batch.py` drives them through the
engine; `scripts/compare_amino_acids.py` aggregates the campaign into a cross-target
table (composition, accessibility, and a discriminator tag: `not_accessible_in_batch`
/ `energetically_fragile` / `inventory_gated` / `robust_accessible`); and
`scripts/plot_nh3_combined.py` builds the NH3-excluded vs NH3-present comparison
(accessibility vs NH3 %, M&P-style). NH3 = 0 is the no-ammonia batch only; NH3 sweeps
start at 0.01 (1% of water). See [`docs/SENSITIVITY_CLI.md`](docs/SENSITIVITY_CLI.md)
for the full campaign workflow. These campaign outputs are committed as part of the
published dataset.
